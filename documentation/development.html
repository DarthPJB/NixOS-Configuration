🔄 Development Workflow

Standard workflow for making changes to the NixOS configuration.

1

Branch Creation

Create a feature branch for your changes.

git checkout -b feature/your-feature-name
2

Local Development

Make your changes to the Nix configuration files.

Edit files in appropriate directories: machines/, environments/, services/, etc.

3

Validation & Formatting

Format and validate your changes before testing.

nix fmt
nix flake check
nix flake show
4

Local Testing

Test your changes on a specific machine configuration.

nixos-rebuild build --flake .#hostname
nixos-rebuild build-vm --flake .#hostname
5

Commit & Push

Commit your changes with descriptive messages.

git add .
git commit -m "feat: descriptive commit message"
git push origin feature/your-feature-name
6

Deployment Testing

Test deployment to target machine.

nix run .#hostname

Monitor services and verify functionality.

7

Permanent Deployment

Apply configuration permanently after verification.

nix run .#hostname -- switch

📏 Code Standards

Coding standards and conventions for the repository.

✅ Nix Coding Standards

Follow these standards for consistent, maintainable code:

  • Use lib.mkIf for conditional configuration
  • Use lib.mkDefault for overridable defaults
  • Use with pkgs; [...] for package lists
  • Use relative imports for local modules
  • Keep modules focused on single responsibilities
  • Use descriptive names matching purpose
  • Validate with nix flake check before committing
  • Format code with nix fmt for consistency
# Example module structure { config, lib, pkgs, ... }: { options = { # Define module options }; config = lib.mkIf config.module.enable { # Implementation here }; }

🚫 Prohibited Practices

These practices are strictly prohibited in this repository:

  • ❌ Docker usage (antithetical to Nix purity)
  • ❌ Cloud provider dependencies
  • ❌ Imperative configurations
  • ❌ Unpinned flake inputs
  • ❌ Committing unencrypted secrets
  • ❌ Direct nixpkgs_unstable input access
